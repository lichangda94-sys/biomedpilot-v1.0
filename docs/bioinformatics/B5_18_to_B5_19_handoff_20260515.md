# B5.18 to B5.19 Handoff

## Completed GSEA Resource Work

B5.17 and B5.18 completed the GSEA gene set resource manager foundation:

- Local GMT import copies user-provided GMT files into the project-local gene set repository and registers them without modifying the original file.
- Reactome pathways can be downloaded from the official Reactome GMT zip and cached locally.
- GO BP, GO CC, and GO MF can be generated from the human GO annotation GAF and cached locally.
- KEGG human pathways can be retrieved through KEGG REST and cached locally.
- All imported or downloaded resources are written to `user_data/bioinformatics/gene_sets/gene_set_registry.json`.
- Registry records include source, URL, license note, version/download date, local path, status, validation summary, gene set count, preview counts, checksum, and file size.
- `selected_for_gsea` is single-select only; selecting one resource clears other selected resources.
- GSEA readiness reads the selected resource and blocks only GSEA preflight / execution when no available resource is selected.
- Unselected gene sets do not block data check, standardization preparation, or DEG preflight.

## Live Network Validation

Real network validation was run after B5.18:

- Reactome pathways: `2855` gene sets.
- GO BP: `11248` gene sets.
- GO CC: `1840` gene sets.
- GO MF: `4812` gene sets.
- KEGG human pathways: `371` gene sets.
- Second calls for all five resources returned `cached=True` and reused local cache.
- After selecting `kegg_hsa_pathways`, GSEA gene set readiness had no blocking errors.

The live run also confirmed that the downloader needs a normal BioMedPilot User-Agent for GO downloads and should use `certifi` CA certificates when available in the local Python runtime.

## Boundaries

- Gene set / GMT resources are GSEA analysis resources, not GEO / TCGA / GTEx data files.
- The gene set registry must not enter the acquisition / recognition ordinary file chain.
- B5.19 should not expand the gene set downloader unless fixing a bug or preserving compatibility.
- B5.19 should not implement real GSEA, DEG, enrichment execution, plotting, or reporting.
- Missing or unselected gene sets must remain non-blocking for data check, standardization preparation, and DEG preflight.

## B5.19 Entry

B5.19 should move into:

```text
Standardized asset repository and extraction router
```

The work should route outputs from recognition / data check into analysis-consumable assets:

- Expression assets.
- Sample metadata assets.
- Group design assets.
- Feature annotation assets.
- Clinical assets.
- Imported result assets.
- Analysis input repository records.

The goal is to separate "what was detected" from "what later analysis can consume" by building a stable standardized asset repository and extraction router. B5.19 should consume existing recognition / readiness / standardization confirmation outputs and produce asset records that downstream preflight and execution stages can rely on.
