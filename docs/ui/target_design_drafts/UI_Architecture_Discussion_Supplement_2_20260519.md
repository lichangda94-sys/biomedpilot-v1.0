## 2. Bioinformatics：数据检查与准备 / Data Check & Preparation

### 2.1 页面定位

“数据检查与准备 / Data Check & Preparation” 是 Bioinformatics 的第三个主页面。

它承接：

```text
数据来源 / Data Source
→ 数据检查与准备 / Data Check & Preparation
```

用户在数据来源页完成数据记录、下载或本地引用后，下一步进入该页面。

### 2.2 页面进入逻辑

用户在数据来源页点击：

```text
数据检查与准备
```

进入第三页。

进入该页面时，系统应已经基于用户选择的数据完成：

- 数据识别；
- 文件类型判断；
- 基础标准化判断；
- 关键信息抽取；
- 文件级识别结果生成。

因此，该页面不再把“运行数据检查”作为默认主按钮。

### 2.3 页面核心任务

该页面的核心任务是：

```text
展示系统识别到的文件级结果，让用户判断识别是否准确，并允许必要的人工纠正。
```

它不是一个只显示 yes / no 的检查页，也不是一个技术诊断页。

用户在此页需要知道：

- 我选进来的每一个文件分别是什么；
- 系统把每个文件识别成了什么类型；
- 哪些文件可以用于分析；
- 哪些文件需要确认；
- 哪些文件缺失或无法读取；
- 当前文件组合是否足以进入下一步。

## 3. 文件级识别原则

### 3.1 必须支持多文件输入

数据检查与准备页必须支持多文件输入场景。

当用户在数据来源页导入多个文件，或选择一个包含多个文件的文件夹时，系统应对每个文件进行：

- 单独识别；
- 单独分析；
- 单独展示；
- 单独确认 / 修改。

### 3.2 禁止的展示方式

页面不得：

- 将多个文件混成一个笼统结果；
- 默认只展示第一个文件；
- 只显示“数据已识别”而不说明是哪一个文件；
- 忽略 supplementary files、metadata、annotation、clinical 文件；
- 把文件级识别结果简化为单一数据集级状态；
- 用一个笼统的数据集摘要替代文件级识别表。

### 3.3 不设置数据集级摘要

本页不设置单独的“数据集级摘要”。

原因：

- 数据集级摘要对用户操作意义有限；
- 容易把多个文件的信息混成一个“看似完整”的摘要；
- 用户真正需要看的是每个文件分别被识别成什么；
- 整体可继续性应作为“检查结论”给出，而不是作为数据集摘要。

核心原则：

```text
用户看文件，系统给结论。
```

## 4. 文件级识别结果表

### 4.1 页面必须显示文件级识别结果表

该表是本页面的核心。

建议字段：

```text
文件名
识别类型
识别摘要
状态
操作
```

示例：

```text
文件级识别结果

| 文件名 | 识别类型 | 识别摘要 | 状态 | 操作 |
|---|---|---|---|---|
| expression_counts.tsv | 表达矩阵 | 20,531 genes × 58 samples；疑似 raw counts | 已识别 | 查看 / 修改 |

| gene_annotation.tsv | 基因注释信息 | 包含 gene_id、gene_symbol、gene_name | 已识别 | 查看 / 修改 |
| clinical.tsv | 临床信息 | 包含 diagnosis、gender、vital_status、survival_time | 已识别 | 查看 / 修改 |
```

### 4.2 文件较多时的处理

如果文件数量较多，应支持：

- 分页；
- 文件类型筛选；
- 按文件类型分组；
- 查看选中文件摘要；
- 修改识别；
- 确认识别。

建议筛选项：

```text
全部
表达矩阵
样本信息
基因注释信息
临床信息
分组信息
其他辅助文件
未识别
已忽略
```

## 5. 文件识别类型

### 5.1 核心识别类型

文件级识别类型应支持：

```text
表达矩阵
样本信息
基因注释信息
临床信息
分组信息
其他辅助文件
未识别
已忽略
```

其中最核心的四类是：

1. 表达矩阵；
2. 样本信息；
3. 基因注释信息；
4. 临床信息。

### 5.2 表达矩阵

用于帮助用户判断当前文件是否确实被识别为表达矩阵。

建议显示：

```text
表达矩阵
状态：已识别 / 未识别 / 需要确认
摘要：检测到 20,531 genes × 58 samples；疑似 raw counts
示例：ENSG00000141510、ENSG00000171862、ENSG00000139618
```

### 5.3 样本信息

用于帮助用户理解当前文件是否包含样本信息，以及样本信息是否与表达矩阵匹配。

建议显示：

```text
样本信息
状态：已识别 / 未识别 / 需要补充
摘要：检测到 58 个样本，样本 ID 与表达矩阵列名匹配
示例字段：sample_id、sample_type、tissue、condition
```

### 5.4 基因注释信息

用于帮助用户判断该文件是否包含基因注释信息，或者后续是否需要平台注释 / ID 转换。

建议显示：

```text
基因注释信息
状态：已识别 / 部分识别 / 未识别
摘要：检测到 gene_id 和 gene_symbol 字段
示例：ENSG00000141510 → TP53
```

如果是 GEO probe：

```text
基因注释信息
状态：需要平台注释
摘要：表达矩阵行名疑似 probe ID，暂未确认对应 gene symbol
下一步：需要平台注释或基因 ID 映射
```

### 5.5 临床信息

用于帮助用户理解当前数据是否包含临床信息，是否可用于后续生存分析、临床关联或样本注释。

建议显示：

```text
临床信息
状态：已识别 / 未识别 / 不适用
摘要：检测到 diagnosis、age、gender、survival_time、vital_status 等字段
示例字段：age_at_diagnosis、gender、vital_status、days_to_death
```

如果没有：

```text
临床信息
状态：未识别
说明：当前数据来源未检测到临床信息。仍可继续进行表达矩阵相关分析。
```

### 5.6 分组信息

允许识别用户单独上传的 group sheet / design table。

注意：

- 可以识别为“分组信息”；
- 但不在此页完成最终分组设计；
- 最终分组与比较设计放到下一页“分组与分析设计”。

- 物种独立卡片；
- 数据值类型独立卡片；
- 分组字段候选大卡片；
- 完整矩阵预览；
- 完整样本表；
- 完整临床表；
- 技术路径；
- manifest；
- source_files；
- raw JSON。

说明：

- 物种可从数据来源或数据集信息判断，不作为本页核心卡片；

- 分组字段候选放入下一页“分组与分析设计”；
- 技术信息进入检查日志。

## 7. 选中文件摘要

用户点击文件级识别表中的某个文件后，应显示该文件的识别摘要。

### 7.1 显示内容

选中文件摘要应包含：

- 文件名称；
- 识别类型；
- 状态；
- 简短摘要；
- 少量示例；
- 确认 / 修改入口。

### 7.2 示例：表达矩阵

```text
文件：expression_counts.tsv
类型：表达矩阵
状态：已识别

摘要：
检测到 20,531 个基因 × 58 个样本。
第一列疑似 gene_id，后续列为样本。
数据值类型：raw counts。

示例：
ENSG00000141510, ENSG00000171862, ENSG00000139618

```

### 7.3 示例：样本信息

```text

样本 ID 与表达矩阵列名匹配。
示例字段：sample_id, sample_type, tissue

操作：
[确认] [修改识别]
```

### 7.4 示例：临床信息


## 8. 整体检查结论

### 8.1 结论定位

整体检查结论根据所有文件级识别结果判断，而不是根据数据集级摘要判断。

它回答：

```text
当前这些文件组合起来，是否可以进入下一步？
如果不能，缺少什么？
如果需要确认，用户需要确认什么？
```

### 8.2 结论状态

整体检查结论包含三种：

```text
可进入下一步
需要确认
需要补充信息
```

### 8.3 可进入下一步

条件示例：

- 文件组合中包含表达矩阵；
- 样本信息已识别，或可从表达矩阵列名生成样本列表；
- 或用户后续可以手动分组；
- 关键文件识别无阻塞错误。

示例文案：

```text
检查结论：可进入下一步

已识别表达矩阵和样本信息，可以进入分组与分析设计。
```

或者：

```text
检查结论：可进入下一步，但需要后续手动分组

已识别表达矩阵，但未识别到样本信息。你仍可进入下一步手动创建分组。
```

### 8.4 需要确认

触发情况示例：

- 多个文件都可能是表达矩阵；
- 表达矩阵候选不唯一；
- 样本 ID 匹配不完全；
- 数据值类型不确定；
- 基因 ID 类型不确定；
- 关键识别项存在歧义。

示例文案：

```text
检查结论：需要确认

系统识别到多个表达矩阵候选，请确认用于后续分析的主表达矩阵。
```

### 8.5 需要补充信息

触发情况示例：

- 缺少表达矩阵；
- 只有临床信息；
- 只有样本信息；
- 只有注释文件；
- 文件无法读取；
- 无可用于分析的表达数据。

示例文案：

```text
检查结论：需要补充信息

当前未识别到表达矩阵。请返回数据来源页补充表达矩阵文件。
```

或者：

```text
检查结论：需要补充信息

当前文件仅包含临床信息，尚不能进行表达矩阵分析。请补充表达矩阵文件。
```

## 12. Bioinformatics：分组与分析设计 / Group & Design

### 12.1 页面定位

“分组与分析设计 / Group & Design” 是后续所有分析任务共享的样本分组与比较设计页面。

它不是 DEG 专属页面。

它应服务于：

- DEG 差异分析；
- GSEA / 富集分析；
- 临床关联分析；
- 生存分析；
- 相关性分析；
- 分组可视化；
- 报告摘要。

### 12.2 页面目标

页面目标是：

- 选择本次分析使用哪些样本；
- 选择或创建样本分组；
- 设定 Group A / Group B；
- 在需要时设定 case / control；
- 设定 paired / unpaired；
- 保存一个或多个 comparison design；
- 为后续多个分析任务提供共享设计输入。

---

## 13. 原始分组字段与分析分组设计

### 13.1 必须区分两层概念

该页面必须区分：

1. 原始分组字段
   数据中已有的字段和值，例如 group、sample_type、condition、treatment、tissue。

2. 分析分组设计
   用户基于原始字段和值创建的分析用分组与比较方案。

系统可以识别和展示原始字段，但不能自动理解所有字段的生物学含义。

### 13.2 原始分组字段

原始分组字段可能包括：

- group；
- sample_type；
- condition；
- treatment；
- tissue；
- batch；
- timepoint；
- dose；
- phenotype。

字段取值可能是：

```text
A / B / C / D
control / drugA / drugB
Primary Tumor / Solid Tissue Normal
0h / 24h / 48h
low / medium / high
```

### 13.3 分析分组设计

分析分组设计是用户为了某次分析而创建的比较方案。

示例：

```text
Design 1: Drug A vs Control
Control = A
Case = B

Design 2: Drug B vs Control
Control = A
Case = C

Design 3: Drug A + Drug B vs Control
Control = A
Case = B + C

Design 4: High dose vs Low dose
Control = B
Case = D
```

## 14. 多个 comparison design

### 14.1 决策

同一个数据集允许保存多个 comparison design。

原因：

- 后续不同分析任务可能使用不同分组；
- 同一组数据可能有多个研究问题；
- 药物处理、多时间点、多剂量、多组织、多阶段数据都可能需要多个比较设计。

### 14.2 示例

同一个项目可以保存：

- Drug A vs Control；
- Drug B vs Control；
- Drug A + Drug B vs Control；
- Tumor vs Normal；
- Stage III-IV vs Stage I-II；
- High expression vs Low expression。

### 14.3 每个 comparison design 应保存的信息

每个 design 应保存：

- 设计名称；
- 使用的原始字段；
- Group A / Control；
- Group B / Case；
- 纳入哪些原始组；
- 排除哪些原始组；
- 纳入样本数；
- 排除样本数；
- paired / unpaired；
- 适用分析任务；
- 备注。

## 15. 原始组到分析组的映射

### 15.1 必须支持的映射能力

页面必须支持：

- 一个原始取值 → 一个分析组；
- 多个原始取值 → 一个分析组；
- 排除某些原始取值；
- 排除某些样本；
- 多组中任选两组进行比较；
- 多个原始组组合成一个 case / treatment group。

### 15.2 示例：A/B/C/D

原始 group 字段：

```text
A, B, C, D
```

用户定义：

```text
Control = A
Drug A = B
Drug B = C
Drug C = D
```

Comparison 1：

```text
Drug A vs Control
Case = B
Control = A
Excluded = C, D
```

Comparison 2：

```text
Drug A + Drug B vs Control
Case = B + C
Control = A
Excluded = D
```

### 15.3 用户手动解释分组含义

当系统识别到 A/B/C/D 这类无法解释的分组值时，应提示：

```text
系统检测到 group 字段包含 A、B、C、D 四个取值，但无法判断其生物学含义。请为这些取值命名，并选择用于比较的分组。
```

## 16. 系统推荐规则

系统推荐只能作为建议，不能自动决定最终分组。

可以提示：

```text
检测到 sample_type 字段，可能可生成 Tumor vs Normal 比较。
```

但必须由用户确认。

系统不得：

- 自动决定 A/B/C/D 的含义；
- 自动选择 case/control；
- 自动合并分组；
- 自动排除样本；
- 自动生成最终 comparison design 而不让用户确认。

## 17. 分组与分析设计页主要区域

建议页面包含：

1. 当前输入数据
   显示上一页确认后的表达矩阵、样本信息、基因注释、临床信息等文件。

2. 样本表
   显示 sample_id、原始字段、用户定义分组、是否纳入分析。

3. 原始分组字段与取值
   展示系统识别到的 group、sample_type、condition、treatment、tissue 等字段及其取值数量。

4. 创建分析分组
   用户为原始字段取值赋予含义，并组合成 Control、Case、Treatment 等分析组。

5. Comparison Design 列表
   保存多个比较设计，供后续不同分析任务使用。

6. 设计检查
   判断是否可进入分析任务，提示样本数不足、缺少对照、paired 信息不完整等问题。

## 19. Bioinformatics：分析任务 / Analysis Tasks

### 19.1 页面定位

“分析任务 / Analysis Tasks” 是 Bioinformatics 的第五个主页面。

它承接：

```text
分组与分析设计
→ 分析任务
```

该页面不是结果页，也不直接堆叠所有运行按钮。

页面目标是：

```text
用户选择想做哪一种分析，系统展示所需输入是否已经准备好，准备好后允许开始分析。
```

### 19.2 页面不做什么

分析任务页不做：

- 展示最终结果；
- 展示完整图表；
- 展示报告；
- 将所有分析按钮平铺堆叠；
- 把未完成或 planned 功能表现为可运行；
- 把 preflight / dry-run 包装成正式统计分析。

结果、图和报告应进入：

```text
结果浏览 / Results
报告导出 / Report
```

## 20. 分析任务页分析类型

页面应按分析类型分区：

1. 差异表达分析 / DEG；
2. 富集分析 / GSEA / ORA；
3. 相关性分析；
4. 临床关联 / 生存分析；
5. 可视化与报告辅助。

其中可视化与报告辅助放在最后，作为分析完成后的后续入口，不作为主要分析任务。

## 21. 每个分析类型区域的统一逻辑

每个分析类型区域应展示：

- 分析目的；
- 所需输入；
- 当前输入准备状态；
- 缺失项；
- 当前是否可以开始；
- 开始分析或准备资源按钮。

核心逻辑：

```text
先检查所需输入是否准备好，再允许开始分析。
```

### 22.1 页面说明

### 22.2 所需输入

- 表达矩阵；
- 样本分组；
- comparison design；
- 数据值类型；
- 样本数量。

### 22.3 展示示例

```text
差异表达分析 / DEG

用于比较两个或多个样本组之间的基因表达差异。

当前输入状态：

- 样本数量：Tumor 50 / Normal 10

```

## 23. 富集分析 / GSEA / ORA

### 23.1 页面说明

用于基于 DEG 结果或排序基因列表进行通路和功能富集分析。

### 23.2 所需输入

项目内输入：

- DEG 结果；
- ranked gene list；
- gene ID 类型；
- 物种。

外部标准资源：

- GO；
- KEGG。

### 23.3 第一版只支持 GO / KEGG

富集分析第一版只提供：

```text
GO
KEGG
```

原因：

- 二者是最常见的基础富集资源；
- 可通过软件联网下载；
- 可缓存到本地后重复调用；
- 不需要复杂账号或权限申请。

### 23.4 暂不接入的资源

暂不接入：

- 需要账号的资源；
- 需要权限申请的资源；
- 商业授权资源；
- 非必要复杂资源；
- 其他当前不必须的 gene set 平台。

注意：

如果未来接入更多资源，需单独讨论授权、账号、缓存和用户提示。

### 23.5 富集分析卡片示例

```text
富集分析 / GSEA / ORA

用于基于 DEG 结果或排序基因列表进行通路和功能富集分析。

当前输入状态：
- DEG 结果：未生成
- Ranked gene list：未提供
- 物种：Human
- Gene ID 类型：Ensembl
- GO 资源：未下载
- KEGG 资源：未下载

状态：需要先完成 DEG 或导入 ranked gene list，并准备 GO / KEGG 资源。

[下载 GO 资源]
[下载 KEGG 资源]
[查看所需输入]
```

## 27. 分析资源与工具

### 27.1 设置中心结构更新

原“外部引擎与模型”扩展为：

```text
外部引擎、模型与分析资源
```

其下包含：

```text
外部引擎、模型与分析资源
├── 图像分析引擎
├── PDF 识别与 OCR 引擎
├── 本地语言模型
├── 云端 AI 服务
└── 分析资源与工具
```

如果 UI 名称过长，也可在界面中简化显示为：

```text
外部能力与资源
```

但文档中使用：

```text
外部引擎、模型与分析资源
```

### 27.2 分析资源与工具定位

“分析资源与工具”用于管理可下载、可缓存、可复用的分析资源和统计工具包。

包括：

- GO 资源；
- KEGG 资源；
- 生信分析工具包；
- Meta 统计工具包；
- 绘图工具包；
- 本地缓存管理。

### 27.3 为什么放在 Settings

这些资源和工具不仅服务 Bioinformatics，未来也可能服务 Meta Analysis 或其他模块。
它们具有共同特征：

```text
第一次下载或安装后，后续本地复用。
```

因此应归入 Settings，而不是只放在 Bioinformatics Settings。

### 27.4 不放在环境检测

GO / KEGG 等资源不是运行环境。
它们是分析资源。

不应放入：

```text
环境检测
```

### 27.5 本地缓存

分析资源与工具应支持本地缓存管理：

- 下载；
- 更新；
- 删除；
- 查看状态；
- 查看版本；
- 清理缓存；
- 默认折叠本地路径。

## 28. 分析任务页中的就地准备入口

虽然分析资源长期管理位置在 Settings，但在分析任务页应保留就地准备入口。

例如在富集分析卡片中：

```text
当前未检测到 GO / KEGG 资源。
你可以现在下载并缓存到本地，后续分析可直接复用。

[下载 GO 资源]
[下载 KEGG 资源]
[进入分析资源设置]
```

原则：

- 用户不需要先去设置中心寻找资源；
- 缺少资源时，在当前任务页直接提示；
- 用户确认后联网下载；
- 下载后缓存到本地；
- 后续复用，不重复下载。

## 29. 安装与首次使用资源下载

### 29.1 不强制安装时下载全部资源

安装或首次启动时不应强制下载全部分析资源。

原因：

- 用户可能只使用 LabTools；
- 用户可能只使用 Meta；
- 用户可能不需要生信资源；
- 不应偷偷联网下载。

### 29.2 首次进入相关分析时提示初始化

当用户首次进入 Bioinformatics 分析任务，尤其是富集分析时，可以提示：

```text
检测到你尚未安装常用生信分析资源。
是否初始化常用分析资源？

可选：
[ ] GO 资源
[ ] KEGG 资源
[ ] 差异分析工具包
[ ] 富集分析工具包

[开始下载]
[稍后再说]
```

原则：

- 用户明确授权；
- 不自动下载；
- 不影响其他模块使用；
- 下载后本地复用。

## 30. Codex 实现原则

后续交给 Codex 实现相关内容时，应遵守：

1. 不进入 Figma；
2. 不做高保真视觉；
3. 不把结果页和分析任务页混合；
4. 分析任务页只显示输入准备、缺失项、状态和开始分析入口；
5. 结果、图、报告进入 Results / Report 页面；
6. 分析任务按 DEG、富集分析、相关性、临床关联 / 生存分析、可视化与报告辅助分区；
7. 富集分析第一版只支持 GO / KEGG；
8. 暂不接入需要账号、权限申请或商业授权的资源；
9. GO / KEGG 可以联网下载并缓存到本地；
10. Settings 中建立“分析资源与工具”；
11. 分析任务页中保留“就地准备资源”入口；
12. 不强制安装时下载全部资源；
13. 用户首次使用相关分析时再提示初始化资源；
14. 不伪装 planned 资源或 planned 分析为已可用；
15. 所有下载和外部资源使用应有用户确认；
16. 具体路径和缓存细节默认折叠或放入开发者诊断。

## 31. 当前确认决策摘要

已确认：

```text
1. 数据检查与准备页以文件级识别为核心，不设置数据集级摘要。
2. 文件级识别表必须展示所有纳入文件，不只展示第一个文件。
3. 文件级识别类型包括表达矩阵、样本信息、基因注释信息、临床信息、分组信息、其他辅助文件、未识别、已忽略。

6. 分组与分析设计页服务所有后续分析任务，不是 DEG 专属页面。
7. 页面必须区分原始分组字段和分析分组设计。
8. 允许同一数据集保存多个 comparison design。
9. 支持一个或多个原始组映射到一个分析组。
10. 系统推荐只能作为建议，不能自动决定最终分组。
11. 分析任务页按分析类型分区，不作为结果页。
12. 分析任务页显示每种分析所需输入、当前准备状态、缺失项和开始分析入口。
13. 可视化与报告辅助放在最后，作为分析完成后的后续入口。
14. 富集分析第一版只支持 GO 和 KEGG。
15. 需要账号、权限或商业授权的资源暂不接入。
16. Settings 中新增“分析资源与工具”，归入“外部引擎、模型与分析资源”。
17. GO / KEGG 资源可联网下载并缓存到本地。
18. 分析任务页提供就地资源准备入口。
19. 安装或首次启动不强制下载全部资源。
20. 首次进入相关分析时提示用户初始化所需资源。
```
