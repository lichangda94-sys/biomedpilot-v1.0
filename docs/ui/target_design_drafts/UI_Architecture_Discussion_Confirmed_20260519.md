## 3. 启动页 / Welcome Page 决策

### 3.1 页面性质

取消当前“伪登录页”作为第一屏。
软件启动后进入轻量启动页 / Welcome Page。
当前版本不要求用户登录，不显示账号密码输入框。
用户测试阶段默认免费使用已开放的本地功能。

### 3.2 启动页职责

启动页只承担以下职责：

- 品牌识别；
- 版本状态说明；
- 本地测试版说明；
- 进入本地工作台；
- 打开最近项目；
- 进入设置；
- 进入关于页面。

启动页不承担以下职责：

- 账号登录；
- 注册账号；
- 忘记密码；
- 购买会员；
- 管理订阅；
- 输入 License；
- 展示价格套餐；
- 展示正式商业账号系统。

### 3.3 账户、订阅与 License

账户、VIP、订阅、License 不放在启动页主流程中。
这些内容统一移入：

```text
设置中心 → 账户与订阅
```

当前阶段该页面应标记为：

```text
Planned / Disabled / 后续版本开放
```

当前测试阶段说明：

```text
当前版本无需登录，已开放的本地功能可免费测试。账户、订阅、VIP 和 License 功能将在后续版本开放。
```


## 6. 关于页面 / About Firefly 决策

### 6.1 页面定位

“关于”页面不做成普通版本信息页，而是做成带有人文色彩的项目说明页。

页面定位为：

```text
About + Project Manifesto + Community Invitation
```

中文可以理解为：

```text
关于萤火虫 / 项目理念 / 写给科研路上的同行 / 共同建设邀请
```

它不仅展示版本号，还要表达：

- 为什么叫萤火虫；
- 为什么做这个软件；
- 它想改变什么；
- 当前为什么免费开放；
- 未来为什么可能会有会员/订阅；
- 希望什么样的人参与测试、反馈、开发和改进。

### 6.2 页面标题

关于页面标题使用：

```text
关于萤火虫 / About Firefly
```

副标题使用：

```text
科研路上的一点微光
```

正文中可以使用：

```text
写给科研路上的同行
```

### 6.3 关于页面内容结构

关于页面建议包含以下部分：

1. 品牌与项目身份；
2. 为什么叫萤火虫；
3. 为什么做这个软件；
4. 当前免费开放的态度；
5. 未来会员 / 订阅的说明；
6. 邀请参与测试、反馈和共同建设；
7. 鲁迅引文作为精神结尾；
8. Developer Preview / 本地测试版 / 非临床和非发表级边界说明。

### 6.4 鲁迅引文

关于页面结尾使用鲁迅语句作为项目精神注脚：

```text
“有一分热，发一分光，即令萤火一般，也可以在黑暗里发一点光，不必等候炬火。”
—— 鲁迅
```

使用方式：

- 不作为页面主标题；
- 不做成口号式大字压迫；
- 作为 About 页面结尾的精神落点；
- 可使用较弱的背景卡片；
- 不借鉴 Sci-Hub 的盗版或对抗性语境；
- 只借鉴“创始人自述 / 项目理念 / 共同建设邀请”的感染力。

### 6.5 关于页面核心文案草稿

```text
关于萤火虫 / About Firefly

科研路上的一点微光

萤火虫不刺眼，但能在黑暗中提供方向。

科研生活常常重复、琐碎、漫长。很多时间被消耗在实验记录、表格整理、数据清洗、文献筛选、图像处理和报告格式中。

这个项目来自真实科研训练中的反复体验：实验记录分散、数据整理重复、生信流程复杂、Meta 分析步骤繁琐，很多实验计算仍然依赖手工表格完成。

这些工作并不总是最有创造性的部分，但它们消耗了大量时间，也容易出错。

我希望萤火虫成为实验台旁的一点微光：
帮助研究者整理实验，
管理数据，
减少机械劳动，
把更多时间留给真正的思考和发现。

当前版本仍处于本地测试阶段。
许多基础工具会尽量免费开放给大家使用，包括实验计算、记录辅助、数据整理和测试级科研工作流。

未来，如果接入云端 AI、外部模型、大规模计算、在线服务或团队协作，一些高级能力可能需要会员、订阅或 License 来维持真实运行成本。

但萤火虫的基础目标不会改变：让科研日常中那些重复、低效、容易出错的工作，变得更清楚、更轻一些。

如果你也是科研路上的学生、实验人员、医生、工程师、开发者或设计者，
如果你也曾被重复表格、实验记录、数据整理和分析流程消耗大量时间，
欢迎参与测试、反馈问题、提出需求，甚至一起改进这个项目。

这个项目希望由真正使用它的人一起塑造。

萤火虫不是为了替代研究者，也不是为了制造一个宏大的自动化幻觉。
它只是希望先把那些重复、琐碎、容易出错的科研日常工作，变得清楚一点、轻一点、可复用一点。

“有一分热，发一分光，即令萤火一般，也可以在黑暗里发一点光，不必等候炬火。”
—— 鲁迅
```

### 6.6 关于页面边界说明

虽然 About 页面应具有人文色彩，但仍必须保留必要边界，避免误导用户。

页面底部或折叠说明中应包含：

```text
当前版本为 Developer Preview / 本地测试版。软件输出仅用于科研流程整理、数据检查、实验记录和内部测试，不构成临床建议、诊断结论、正式统计结论或发表级报告。
```

还应包含：

```text
当前版本以本地使用为主。AI 与外部引擎功能默认关闭或需用户自行配置。
```

这些说明可以放在页面下方的“小字说明”或“版本与边界”折叠区中，不抢占人文主叙事。

## 7. Dashboard / 全局工作台决策

### 7.1 页面定位

Dashboard 是用户点击启动页“进入本地工作台”后的第一主界面。
它不是营销页，不是项目中心，不是设置中心，也不是测试说明页。
它的主要职责是让用户快速进入三大主模块，并看到最近项目和当前本地状态。

### 7.2 顶部区域

显示：

- 萤火虫 / Firefly
- Developer Preview · 本地测试版
- 设置 / 关于，小入口

不显示：

- 账户订阅提示；
- 购买 / 升级入口；
- 大量 testing 说明；
- 图标资源状态；
- 技术路径或开发者诊断信息。

### 7.3 主模块区

Dashboard 主模块区放置三个主模块：

1. Bioinformatics / 生信分析
2. Meta Analysis / Meta 分析
3. LabTools / 实验工具

三个模块作为同级主入口展示。

### 7.4 最近项目区域

保留轻量最近项目区域。

显示字段：

- 项目名称；
- 模块类型；
- 最近修改时间；
- 打开。

该区域不是完整 Project Center，不做复杂项目管理。

### 7.5 底部状态栏

底部左侧显示：

- 本地环境可用；
- 外部引擎可选配置。

底部右侧显示：

- 测试反馈入口。

### 7.6 Dashboard 不显示内容

Dashboard 不显示：

- 图标资源状态；
- 账号 / VIP / 订阅 / License；
- Project Center；
- Data Center；
- Task Center；
- Report Center；
- Environment Center；
- Packaging；
- manifest / registry / cache path 等技术细节。

### 7.7 Dashboard 结构草稿

```text
Dashboard / 全局工作台

顶部：
- 萤火虫 / Firefly
- Developer Preview · 本地测试版
- 设置 / 关于，小入口

主模块区：
[Bioinformatics / 生信分析]
[Meta Analysis / Meta 分析]
[LabTools / 实验工具]

最近项目：
- 项目名称
- 模块类型
- 最近修改时间
- 打开

底部左侧：
- 本地环境可用
- 外部引擎可选配置

底部右侧：
- 测试反馈入口
```

## 8. Dashboard 主模块卡片决策

### 8.1 卡片定位

Dashboard 主模块卡片展示产品目标能力，而不是只展示当前已完成能力。
目的是让测试用户、潜在贡献者和同行理解每个模块未来希望实现的完整方向，并知道可以在哪些地方提出建议或参与改进。

但所有卡片必须统一标记：

```text
状态：测试中
```

按钮统一为：

```text
进入
```

卡片不展示过多限制说明。
具体模块的当前接入状态、未完成功能、testing 边界，应在进入模块后的模块首页中说明。

### 8.2 Bioinformatics / 生信分析 卡片文案

```text
Bioinformatics / 生信分析

面向 GEO、TCGA、GTEx 与本地表达矩阵的数据分析工作台，帮助用户完成数据导入、检查、分组、差异分析、富集分析、结果整理和报告生成。

能力方向：
GEO / TCGA / GTEx · 数据检查 · 分组设计 · DEG · GSEA · 结果报告

状态：测试中

[进入]
```

```text
能力方向：
```
